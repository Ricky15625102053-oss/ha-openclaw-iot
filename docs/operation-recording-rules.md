# 操作记录与解释规范

更新时间：2026-07-09

## 1. 固定要求

本项目后续每次操作都需要记录下来，并给出详细、可解释的说明。

每条记录必须说明：

- 为什么要这样做。
- 本次操作的目的是什么。
- 具体怎么做。
- 执行结果是什么。
- 结果代表什么。
- 下一步应该做什么。

重复出现、原因相同、处理方式相同的问题不需要反复记录。只需要在原有问题记录中补充“再次出现”和最新状态，避免文档膨胀。

## 2. 记录范围

需要记录的操作：

| 类型 | 示例 |
| --- | --- |
| 项目代码修改 | 新增脚本、修改配置、更新文档 |
| Git 操作 | 提交、推送、推送失败、远程状态变化 |
| 服务器操作 | 安装 Docker、启动 Home Assistant、检查端口 |
| 阿里云控制台操作 | 安全组放行端口、重置密码、Workbench 登录 |
| Home Assistant 操作 | 初始化账号、生成 Token、接入设备 |
| OpenClaw 操作 | 配置 Skill、调用 HA API、实现安全规则 |
| 问题排查 | 命令卡住、端口打不开、网络失败、容器启动失败 |

不需要重复记录的情况：

- 同一个 GitHub 网络失败原因已经记录，再次失败只更新最后一次时间和状态。
- 同一个命令因为同样原因卡住，不需要新建一整段记录。
- 同一个安全组端口已经确认开放，不需要每次都重复说明。


## 4. 标准记录模板

每次推进项目时，使用下面格式：

```text
## 日期：2026-xx-xx

### 本次目标

说明这次要解决什么问题。

### 为什么要做

解释这样做的原因，以及它和项目目标的关系。

### 具体怎么做

1. 写清楚在哪台电脑或哪台服务器上操作。
2. 写清楚执行了什么命令或修改了什么文件。
3. 写清楚每一步的作用。

### 执行结果

记录关键输出、文件变化、提交号、访问地址或错误信息。

### 结果解释

说明这个结果代表成功、失败、部分完成，还是需要继续排查。

### 是否属于重复问题

如果是重复问题，写清楚已在哪个文档记录过，本次只补充最新状态。

### 下一步

写清楚下一次应该从哪里继续。
```

## 5. 敏感信息要求

记录操作时不能写入真实敏感信息，包括：

- 服务器密码。
- SSH 私钥。
- GitHub Token。
- Home Assistant Long-Lived Access Token。
- OpenClaw API Key。
- 大模型 API Key。
- 米家账号密码。

可以记录“已经生成”“保存到本机密码管理器”“用于 OpenClaw 调用 HA API”，但不能记录真实内容。
